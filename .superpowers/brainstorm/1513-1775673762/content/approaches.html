Como implementar

3 abordagens — cada uma com trade-offs diferentes

A

CustomPainter turbinado recomendado

Reescrever o schematic com geometria fiel ao mapa oficial: polylines com múltiplos segmentos, coordenadas calibradas manualmente para todas as ~20 linhas. O painter ganha: labels de estações, marcadores de integração (círculo branco com bordas coloridas), modo dark adaptativo.

Prós

  • Animações suaves (já existe)
  • Dados em tempo real integrados
  • Zoom/pan fluido com InteractiveViewer
  • Dark mode perfeito

Contras

  • Coordenadas precisam ser digitalizadas manualmente
  • Trabalho de calibração das ~20 linhas
B

SVG embutido + overlay interativo

Usar o SVG oficial do mapa da CPTM/Metro como imagem base, renderizar por cima overlay de interatividade (taps, indicadores de lotação) usando Stack + Positioned.

Prós

  • Visual 100% idêntico ao impresso
  • Zero esforço de calibração

Contras

  • Dark mode impossível (SVG é imagem fixa)
  • Animações da linha selecionada muito difíceis
  • Indicadores de lotação desalinhados ao zoom
  • SVG oficial não disponível gratuitamente
C

flutter_map + tiles geográficos

Usar coordenadas geográficas reais (lat/lon) e renderizar sobre mapa de fundo (Mapbox/OSM com estilo monocromático). Estações são markers sobrepostos.

Prós

  • Posições geograficamente precisas
  • Escala automática

Contras

  • Mapa geográfico ≠ mapa esquemático (confuso)
  • Requer API key + custos de tiles
  • Perde a identidade visual do mapa de metrô
  • Animações por linha muito complexas

Em paralelo ao mapa: design system premium aplicado em todo o app — typography Inter/SF Pro, espaçamentos consistentes, dark mode completo com ThemeData.dark().
